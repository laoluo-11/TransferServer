# Bumi 中间层开发计划

## 1. 项目目标

本项目目标是为 Bumi 机器人构建一套可落地的智能中间层，使机器人具备以下能力：

- 统一接入云端大模型或智能体
- 统一编排运动、语音和任务执行
- 统一管理状态上报、日志、告警
- 通过本地安全层保证控制可靠

第一阶段不追求全自主机器人，而是优先实现“稳定可控的智能演示能力”。

## 2. MVP 目标

第一阶段 MVP 成功标准：

- 云端可以向机器人下发结构化任务
- 机器人可以执行 `move`
- 机器人可以执行 `gesture`
- 机器人可以执行 `play_teach`
- 机器人可以执行 `speak`
- 支持中断任务
- 支持状态上报和错误上报
- 支持低电量拒绝运动任务

如果以上能力全部跑通，就已经具备对外演示和内部迭代基础。

## 3. 项目边界

本阶段纳入范围：

- Bumi `Highcontrol` 二次封装
- Robot Agent
- 云端中间层服务
- 任务与状态协议
- 语音播报
- 安全规则
- 基础管理接口

本阶段不纳入范围：

- `Lowcontrol` 关节级智能控制
- 完整自主导航
- 复杂视觉闭环跟随
- 多机器人协同调度
- 复杂权限系统

## 4. 项目角色建议

建议最小团队配置：

- `1` 名机器人控制开发
  - 负责 Bumi SDK 对接和本地控制封装
- `1` 名后端开发
  - 负责云端服务、协议、数据库、任务接口
- `1` 名应用编排开发
  - 负责 Robot Agent、状态机、语音和智能体接入
- `1` 名测试或兼任测试
  - 负责联调、用例、回归和日志分析

如果人员有限，至少需要：

- `1` 名偏机器人方向开发
- `1` 名偏后端和编排方向开发

## 5. 总体里程碑

建议分成四个阶段推进。

### 阶段 1：跑通机器人本地控制链路

目标：

- 本地代码可稳定控制 Bumi 执行基本技能

输出：

- `move`
- `stop`
- `gesture`
- `play_teach`
- 状态读取

### 阶段 2：完成 Robot Agent

目标：

- 本地代理能接收远程任务并执行

输出：

- 长连接
- 状态机
- 安全层
- 任务结果上报

### 阶段 3：完成云端中间层

目标：

- 云端可创建任务、记录日志、查看状态

输出：

- REST API
- 任务调度
- 状态存储
- 告警存储

### 阶段 4：接入云端智能体

目标：

- 大模型可通过技能调用驱动机器人

输出：

- 工具调用封装
- 多轮任务编排
- 演示流程

## 6. 分阶段详细任务

## 6.1 阶段 1：机器人本地控制封装

### 目标

把 Bumi SDK 封装成一组稳定、可测试、可复用的本地能力接口。

### 任务清单

1. 阅读 Bumi 交付文档并确认 `Highcontrol` 可用动作集
2. 建立本地测试工程
3. 封装 `move(x, yaw, duration_ms)`
4. 封装 `stop()`
5. 封装 `gesture(name)`
6. 封装 `play_teach(index)`
7. 封装状态读取接口
8. 增加本地日志输出
9. 增加超时自动停止逻辑

### 验收标准

- 本地命令可以稳定触发移动
- 本地命令结束后机器人会自动停止
- `PLAYTEACH` 不会重复触发卡住
- 连续执行 `gesture + speak` 不会让控制线程失控
- 所有执行流程都有日志

### 风险点

- SDK 实际行为与文档描述存在差异
- `workmode` 切换时机可能和预期不同
- 某些动作不能并行执行

### 应对策略

- 每个动作先做单独验证
- 记录模式切换日志
- 严格限制第一版可执行动作集合

## 6.2 阶段 2：Robot Agent

### 目标

实现机器人本地执行代理，把本地控制能力升级为远程受控服务。

### 任务清单

1. 实现 Agent 启动流程
2. 实现配置加载
3. 实现 WebSocket 长连接
4. 实现 `agent_hello`
5. 实现 `heartbeat`
6. 实现任务接收和解析
7. 实现 `task_ack`
8. 实现任务状态机
9. 实现 `task_event`
10. 实现 `task_result`
11. 实现安全规则模块
12. 实现中断任务逻辑
13. 实现本地状态缓存
14. 实现异常捕获与恢复

### 验收标准

- Agent 启动后自动连接云端
- 云端断开后 Agent 能自动重连
- 收到任务后会先确认再执行
- 运动任务被中断时会进入安全停止
- 低电量情况下移动任务会被拒绝

### 风险点

- 长连接稳定性不足
- 任务重复下发导致重复执行
- 异常中断后状态不一致

### 应对策略

- 引入任务幂等
- 引入本地任务状态缓存
- 所有异常默认进入 `safe_stop`

## 6.3 阶段 3：云端中间层

### 目标

构建一个可管理单机器人任务、状态和日志的云端服务。

### 任务清单

1. 初始化云端服务项目
2. 设计数据库表
3. 实现机器人注册信息管理
4. 实现任务创建接口
5. 实现状态查询接口
6. 实现任务查询接口
7. 实现中断接口
8. 实现 WebSocket 会话管理
9. 实现消息转发
10. 实现任务持久化
11. 实现告警持久化
12. 实现基础鉴权
13. 实现审计日志

### 验收标准

- 可通过 API 创建技能任务
- 可查询机器人最新状态
- 可查询任务执行结果
- 可中断当前任务
- 云端断开后能正确标记离线状态

### 风险点

- 云端状态与机器人本地状态不同步
- API 创建任务过快导致排队和抢占冲突

### 应对策略

- 增加状态时间戳和 trace_id
- 明确优先级和抢占规则

## 6.4 阶段 4：智能体接入

### 目标

让大模型通过工具调用方式控制机器人，而不是输出自由控制文本。

### 任务清单

1. 定义技能工具列表
2. 定义工具参数约束
3. 实现智能体调用云端任务接口
4. 实现执行结果回填
5. 实现多轮任务编排
6. 实现演示场景脚本
7. 实现基础知识问答

### 验收标准

- 用户发出自然语言请求后，智能体能选择正确技能
- 智能体不会输出非法动作参数
- 动作执行结果可回到会话上下文
- 演示场景可连续完成

### 风险点

- 大模型输出参数不稳定
- 智能体连续调用过多技能
- 任务打断时上下文混乱

### 应对策略

- 技能参数严格白名单校验
- 每轮只允许有限数量技能调用
- 会话层保存任务状态和最近结果

## 7. 推荐时间安排

下面是一份适合小团队的建议排期，默认 `4` 到 `6` 周。

### 第 1 周

- 读文档
- 明确可用技能
- 建本地测试工程
- 跑通 `move`、`stop`

### 第 2 周

- 跑通 `gesture`、`play_teach`
- 完成状态读取
- 增加日志和超时保护

### 第 3 周

- 开发 Robot Agent
- 打通 WebSocket
- 跑通任务接收、确认、执行、结果上报

### 第 4 周

- 开发云端中间层
- 跑通 REST API
- 跑通任务创建和状态查询

### 第 5 周

- 加入语音播报
- 加入中断逻辑
- 加入低电量与安全规则

### 第 6 周

- 接入智能体
- 做联调
- 准备演示脚本

如果资源紧，可以先把第 `5` 周和第 `6` 周合并，但不建议压缩安全规则开发。

## 10. 验收清单

项目阶段性验收建议用下面这份最小清单。

### 功能验收

- 能通过 API 下发 `move`
- 能通过 API 下发 `gesture`
- 能通过 API 下发 `speak`
- 能查询状态
- 能中断任务
- 能返回结果

### 稳定性验收

- 连续执行 `20` 次任务不崩溃
- 网络闪断后 Agent 能恢复连接
- 超时任务能自动停机
- 重复任务不会重复执行

### 安全验收

- 低电量时移动任务被拒绝
- 中断命令能立即停机
- 非白名单技能被拒绝
- 机器人异常状态时不执行新运动任务

## 11. 测试计划

建议最少分三层测试。

### 单元测试

- 协议字段校验
- 技能参数校验
- 错误码映射
- 状态机流转

### 集成测试

- 云端到 Agent 的任务链路
- Agent 到 SDK 的动作链路
- 中断和超时链路

### 现场联调测试

- 真机移动测试
- 真机动作切换测试
- 真机语音播报测试
- 异常恢复测试

## 12. 风险清单

### 技术风险

- Bumi SDK 行为边界不完全透明
- 任务执行时机依赖 `workmode`
- 网络波动影响用户体验

### 项目风险

- 早期想做的功能过多
- 没有先建立技能边界
- 没有先做好安全兜底

### 组织风险

- 机器人端和云端协议不同步
- 缺少统一日志和 trace_id

## 13. 风险缓解建议

### 技术上

- 先做小技能集
- 先做单机器人
- 先做可观察日志

### 管理上

- 每周固定一次协议评审
- 每周固定一次真机联调
- 所有任务必须有验收标准

## 14. 演示场景建议

建议先做三个可重复演示的场景。

### 场景 1：迎宾

- 机器人挥手
- 机器人自我介绍
- 机器人回答一个简单问题

### 场景 2：讲解

- 机器人播放一段欢迎词
- 机器人转向目标区域
- 机器人做一个示意动作

### 场景 3：人工接管

- 机器人执行移动
- 中途触发中断
- 机器人停止并播报提示

这三个场景足以验证控制、语音、打断、安全四条核心链路。

## 16. 建议先做的三件事

如果准备正式开工，建议先按这个顺序启动：

1. 明确技能清单和参数范围
2. 在机器人侧封装并验证 `move`、`stop`、`gesture`
3. 定义共享协议模型并跑通 Agent 心跳

只要这三件事做成，后续云端和智能体接入就会顺很多。
